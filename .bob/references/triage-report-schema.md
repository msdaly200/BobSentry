# Bob-Sentry Triage Report Schema

> **Purpose:** This document defines the canonical structure for all triage reports
> generated by Bob-Sentry. Every report MUST follow the section order and heading names
> defined here. Section content varies by session type (see §3), but section presence
> and ordering are non-negotiable.
>
> **Location contract:** Reports are always saved to:
> `.bob/reports/<attack-class-folder>/<issue-number>/triage-<issue-number>-<YYYY-MM-DD>.md`
>
> **This schema is the single source of truth.** When the SKILL.md and AGENTS.md
> reference report structure, they defer to this document.

---

## 1. Session Types

Every triage session falls into exactly one type. The type is determined by whether
sandbox execution completed. It controls which sections are **required** vs **omitted**.

| Type | Trigger | Sandbox run? |
|------|---------|-------------|
| `plan-only` | Code mode produced scripts but Agent mode was not invoked | No |
| `executed` | Agent mode ran all phases to completion | Yes |
| `escalated` | Agent mode halted on a `500`, setup failure, or novel pattern | Partial / No |

The session type must appear in the front-matter as `**Session Type:**`.

## 2. Front-Matter Block

The report MUST open with this front-matter block. All fields are required.

```markdown
# Security Triage Report — Issue #{{ISSUE_NUMBER}}

**CVE:** {{CVE_ID_OR_NONE}}
**Title:** {{ISSUE_TITLE}}
**Session Type:** plan-only | executed | escalated
**Verdict:** {{VERDICT_EMOJI}} {{VERDICT_TEXT}}
**Severity:** {{SEVERITY}}
**Analyst:** Bob-Sentry (automated triage)
**Triage Date:** {{YYYY-MM-DD}}
**Keycloak Version Tested:** {{VERSION_OR_NA}}
**Report Path:** `.bob/reports/{{REPORT_FOLDER}}/{{ISSUE_NUMBER}}/triage-{{ISSUE_NUMBER}}-{{DATE}}.md`
**Session Metadata:** `.bob/reports/{{REPORT_FOLDER}}/{{ISSUE_NUMBER}}/context.json`
```

**Verdict emoji + text values:**

| Verdict | Emoji + Text |
|---------|-------------|
| confirmed | `✅ CONFIRMED — VULNERABILITY ACTIVE` |
| not-vulnerable | `🟢 NOT VULNERABLE — PATCH VERIFIED` |
| escalated | `⚠️ ESCALATED — REQUIRES HUMAN REVIEW` |
| plan-only | `📋 PLAN ONLY — SANDBOX NOT YET EXECUTED` |

## 4. Section Specifications

### 4.1 Executive Summary
`Type: prose paragraph(s)`

A 2–5 sentence plain-English description of:
- What the vulnerability is
- Who can exploit it (actor + required permission)
- What the impact is
- Whether it was confirmed in a live sandbox (for `executed` sessions)

For `plan-only` sessions this section is optional. When present it should summarise
the threat analysis, not the execution.

---

### 4.2 Phase 1 — Guardrails

`Type: table`

All 8 rules MUST appear. Status values: `✅`, `✅ N/A`, `⚠️ PENDING`, `❌ VIOLATION`.

```markdown
| Rule | Description | Status |
|------|-------------|--------|
| Rule 1 | Network isolation — all requests to localhost only | ✅ |
| Rule 2 | Mock IdP — WireMock used / N/A | ✅ N/A |
| Rule 3 | Response classification applied | ✅ |
| Rule 4 | Only approved test credentials used | ✅ |
| Rule 5 | Secret scan passed on both scripts before execution | ✅ |
| Rule 6 | Container cleanup: completed / pending / N/A | ✅ Pending |
| Rule 7 | GitHub accessed read-only only | ✅ |
| Rule 8 | All runtime files written to /tmp/keycloak-triage/ | ✅ |
```

`Type: table`

All rows below are required. Omit only rows explicitly marked optional.

```markdown
| Field | Value |
|-------|-------|
| **Issue Number** | [#{{ISSUE_NUMBER}}](https://github.com/keycloak/keycloak/issues/{{ISSUE_NUMBER}}) |
| **Assigned CVE** | {{CVE_ID_OR_NONE}} |
| **Matched CVE Pattern** | {{PATTERN_CVE}} — {{PATTERN_DESCRIPTION}} |
| **Attack Type** | {{ATTACK_TYPE}} |
| **Attack Class** | Class {{N}} — {{CLASS_NAME}} |
| **Affected Component** | {{COMPONENT}} |
| **Confidence** | HIGH / MEDIUM / LOW (→ upgraded to HIGH after live confirmation) |
| **Novel Pattern** | YES / NO |
| **Novel Sub-Variant** | YES / NO — (optional, omit if not applicable) |
| **Affected Version Band** | {{VERSIONS}} |
| **Keycloak Version Tested** | {{VERSION}} (image: `quay.io/keycloak/keycloak:{{VERSION}}`) |
| **CVSS v3.1 Base Score** | **{{SCORE}}** |
| **CVSS v3.1 Vector** | `{{VECTOR}}` |
| **Severity** | **{{SEVERITY}}** |
```

Follow the table with a blockquote CVSS rationale for any score above 7.0:
```markdown
> **CVSS rationale:** [one sentence explanation of each significant dimension]
```

### 4.4 Phase 2 — Extraction & Pattern Matching

`Type: two sub-sections`

#### 4.4.1 Extraction Pass
A table of raw technical terms extracted from the issue body:

```markdown
| Element | Value |
|---------|-------|
| Issue title | ... |
| Actor / prerequisite | ... |
| Attack vector | ... |
| Target / impact | ... |
| Bypass mechanism | ... |
| Fix versions | ... |
```

Add rows for any issue-type-specific fields (e.g., `Entry point`, `Attack header` for DoS;
`SSRF payload` for SSRF; `Mapper type` for token-injection issues).

#### 4.4.2 Pattern Match Results
A fenced code block listing all registered CVE patterns scored against the issue,
highest score first. Always list all registered patterns, even zero-scores:

````markdown
```
PATTERN MATCH RESULTS:
1. {{CVE_ID}} ({{ATTACK_TYPE}}) — score: N/M keywords matched
   Matched: [keyword, keyword, ...]
   Note: [one line reasoning]

2. ...
```
````

Follow with a **Classification** paragraph explaining why the top match was selected
(or why it was overridden to a different class / novel pattern).

---

### 4.5 Phase 2 — Root Cause Analysis

`Type: prose + code block`

Required for `executed` sessions. Optional for `plan-only`.

Structure:
1. **Vulnerable code area** — file paths + method names (fenced block or table)
2. **Root cause** — one paragraph of plain-English explanation
3. **Architectural anti-pattern** — reference the relevant anti-pattern from the
   CVE history Cross-Pattern Analysis section (e.g., "Authorization asymmetry — Class 10")

For **third-party / dependency CVEs** (novel-pattern sessions where the root cause is
in an upstream library), replace item 3 with:
- **Keycloak amplifier** — what Keycloak configuration makes the upstream bug reachable

### 4.6 Phase 3 — Sandbox Configuration

`Type: table`

```markdown
| Parameter | Value |
|-----------|-------|
| Image | `quay.io/keycloak/keycloak:{{VERSION}}` |
| Container name | `keycloak` |
| Host port | `127.0.0.1:{{HOST_PORT}} → container:8080` |
| Realm | `triage-realm` |
| Client | `{{CLIENT_ID}}` ({{CLIENT_DESCRIPTION}}) |
| {{Role-specific rows}} | ... |
| {{CVE-specific rows}} | ... |
```

Add rows for every entity created in `setup_realm.py` that is relevant to the exploit
(delegated admin, victim user, injected mapper, CIBA endpoint, backchannel URL, etc.).

For **novel-pattern DoS sessions**, also include:

```markdown
| Exploit parameters | {{e.g., baggage header size, request count}} |
| Feature flag required | {{e.g., --tracing-enabled=true}} |
```

### 4.7 Phase 3 — Script A: Setup (`setup_realm.py`)

`Type: fenced code block or prose outline`

**For `executed` sessions:** Show the actual file path and a note that the script
is saved alongside this report. Do not inline the full script body — it is already
in `setup_realm.py` next to this report.

```markdown
Script saved to: `.bob/reports/{{REPORT_FOLDER}}/{{ISSUE_NUMBER}}/setup_realm.py`

Setup steps performed:
1. {{Step description}}
2. ...
```

**For `plan-only` sessions:** Show the full annotated pseudocode outline so an
engineer can review the intended setup before execution.

---

### 4.8 Phase 3 — Script B: Exploit (`exploit_test.py`)

`Type: fenced code block or prose outline`

Same rule as §4.7. For `executed` sessions: file reference + numbered step list.
For `plan-only` sessions: full annotated pseudocode including Rule 3 classification
logic and expected responses for both vulnerable and patched branches.

---

### 4.9 Phase 3 — Execution Logs

`Type: fenced code blocks` *(executed sessions only)*

Two sub-sections, each containing the captured `stdout` from the script run:

```markdown
### Script A Execution Log

```
[SETUP] ...
RESULT: {...}
```

**Exit code:** 0 ✅

### Script B Execution Log

```
[EXPLOIT] ...
[ASSERT] ...
RESULT: {...}
```

**Exit code:** {{CODE}} — {{MEANING}}
```

For **SSRF sessions** where Docker log evidence is the primary confirmation, add a
third sub-section:

```markdown
### Docker Log Evidence (primary SSRF confirmation)

```
ERROR [BackchannelAuthenticationCallbackEndpoint] ...
```
```

---

### 4.10 Phase 3 — HTTP Evidence

`Type: table + optional JSON block` *(executed sessions only)*

```markdown
| Step | Endpoint | Actor | Expected | Actual | Verdict |
|------|----------|-------|----------|--------|---------|
| {{Step name}} | `{{METHOD}} {{PATH}}` | `{{actor}}` | `{{expected}}` | **`{{actual}}`** | 🔴/🟢 |
```

For token-injection vulnerabilities: follow the table with a fenced JSON block showing
the decoded JWT claim that confirms the injection:

```json
{
  "realm_access": {
    "roles": ["realm-admin", ...]
  }
}
```

For SSRF vulnerabilities: follow the table with the Docker log line that is the
primary confirmation (see §4.9 note).

For DoS vulnerabilities: follow the table with the health check result and any
container log excerpt showing JVM / Vert.x saturation.

### 4.12 Phase 4 — Bug Triage Checklist

`Type: 6-section checkbox list`

Run through all 6 sections of the Expanded Bug Triage Checklist from
[`keycloak-cve-history.md`](.bob/references/keycloak-cve-history.md). Mark each
item `[✓]`, `[?]`, or `[ ]`. Include a one-line rationale after each non-`[ ]` item.

```markdown
### 1. Input Validation
- `[✓]` User-controlled input used in URLs, external requests, or file paths? — {{reason}}
- `[ ]` Redirect URIs... — Not applicable
...

### 2. Authorization Consistency
...
```

For issue types where entire sections are not applicable (e.g., a DoS issue has
nothing relevant in §2 Authorization Consistency), collapse the section to a single
line: `- [ ] All items — Not applicable ({{reason}})`

---

### 4.13 Phase 4 — Suggested Fix / Remediation

`Type: prose + code block(s)`

**For Keycloak logic vulnerabilities** (`plan-only` or `executed`):
- Provide a **fix sketch** — a Java snippet showing the missing or incorrect guard,
  with a comment indicating where it belongs
- List related files that must be audited for the same pattern (e.g., SAML counterpart
  of an OIDC mapper, PUT path in addition to POST)

**For third-party / dependency CVEs** (novel-pattern DoS, upstream library bugs):
- **Primary fix:** the dependency upgrade required
- **Interim mitigations:** ordered list of workarounds that reduce exposure until
  the primary fix ships (feature flags, proxy rules, configuration bounds)

In both cases, conclude with a **Related files to audit** list of clickable file links.

### 4.14 Phase 4 — Action Items

`Type: table`

```markdown
| Priority | Action | Owner |
|----------|--------|-------|
| 🔴 P0 | {{immediate blocking action}} | {{team or person}} |
| 🔴 P1 | {{critical near-term action}} | {{team}} |
| 🟡 P2 | {{important but not blocking}} | {{team}} |
| 🟢 P3 | {{good to have}} | {{team}} |
```

Priority definitions:
- **P0** — must complete before any deployment to affected environment
- **P1** — must complete in the current sprint
- **P2** — schedule in the next sprint
- **P3** — backlog; include in next retrospective

Always include at minimum:
- One P0 or P1 fix action pointing at the specific file/method
- One P1 test action (integration test that would catch a regression)
- One P2 or P3 audit action (related code paths to inspect)

### 4.15 Phase 4 — Session Metrics

`Type: table` — **Required for `executed` sessions; optional for others.**

This section serves two purposes:

1. It is the **per-issue record** inside the triage report itself.
2. It is the **source data** for the running totals in
   [`.bob/reports/metrics-summary.md`](../reports/metrics-summary.md).
   After writing the triage report, Bob-Sentry MUST update `metrics-summary.md`
   with this session's row. If the same issue was triaged before, the previous row
   MUST be replaced with the latest session's data (re-triages supersede).

**Required fields:**

```markdown
## Phase 4 — Session Metrics

| Metric | Value |
|--------|-------|
| Triage date | {{YYYY-MM-DD}} |
| Keycloak version tested | {{VERSION}} |
| Attack class | {{e.g., AuthZ Bypass — FGAP (Class 10)}} |
| CVE / matched pattern | {{CVE_ID_OR_NONE}} / {{MATCHED_CVE_PATTERN}} |
| Sandbox start | Keycloak ready on attempt {{N}} |
| Script A exit | {{CODE}} ({{meaning}}) |
| Script B exit | {{CODE}} ({{meaning}}) |
| Bob-Sentry elapsed time | ~{{N}} minutes |
| Script iterations | {{N}} (Script A: {{a}}, Script B: {{b}}) |
| Primary evidence | {{e.g., HTTP 201 on mapper POST / Docker log line}} |
| Estimated manual triage time | ~{{N}} hours |
| Estimated time saved | ~{{N}} hours |
| Token cost | est. {{RANGE}} tokens |
| Secret scan | PASS / FAIL |
| Container cleanup | completed / pending |
```

**Field guidance:**

| Field | How to populate |
|-------|----------------|
| `Bob-Sentry elapsed time` | Wall-clock minutes from first guardrails read to report written. If not tracked precisely, use ~15 min for a standard 2-script session, ~27 min for a first-run session with multiple iterations. |
| `Script iterations` | Total number of script revisions across both scripts (Script A + Script B runs, including retries). |
| `Estimated manual triage time` | Time a senior engineer unfamiliar with the code path would need to replicate the triage manually. Use ~3 hrs as the default for a complex authorization flaw; ~1.5 hrs for SSRF / dependency CVE; ~4 hrs for novel patterns. |
| `Estimated time saved` | `manual estimate − Bob-Sentry elapsed time (converted to hours)`. |
| `Token cost` | Use `est. 80k–120k` for simple sessions; `est. 100k–150k` for standard sessions; `est. 150k–300k` for first-run or multi-iteration sessions. |
| `Attack class` | Matches the Attack Class column in `metrics-summary.md`. Use the short form from the Attack Class Distribution table. |

**Dual-write rule:** The `metrics-summary.md` Per-Session Log row for this issue MUST
contain the same values as this table. The mapping is:

| `metrics-summary.md` column | Source field in this table |
|-----------------------------|---------------------------|
| Date | Triage date |
| Attack Class | Attack class |
| CVE | CVE / matched pattern (CVE ID only) |
| Severity | From §4.11 Severity Assessment |
| Status | From Verdict (front-matter) |
| Elapsed (min) | Bob-Sentry elapsed time |
| Script Iters | Script iterations (total) |
| Manual Est. (hrs) | Estimated manual triage time |
| Time Saved (hrs) | Estimated time saved |
| Tokens | Token cost |

## 5. Heading Conventions

| Rule | Detail |
|------|--------|
| H1 | Document title only — `# Security Triage Report — Issue #{{N}}` |
| H2 | Phase sections — `## Phase 1 — Guardrails`, `## Phase 2 — Threat Profile`, etc. |
| H3 | Sub-sections within a phase — `### Extraction Pass`, `### Script A Execution Log`, etc. |
| H4 | Nested items only — use sparingly |
| Phase prefix | Sections 2–4 use `Phase 2 —`; sections 6–10 use `Phase 3 —`; sections 11–16 use `Phase 4 —` |
| No phase prefix | Front-matter block and footer have no phase prefix |

## 6. Issue-Type Variations

The schema is fixed; the **content** of certain sections varies by issue type.
This table documents what changes — nothing else may differ from the base schema.

| Issue Type | Section variations |
|------------|--------------------|
| **Admin API / privilege escalation** (e.g., FGAP, TOCTOU, mapper injection) | HTTP Evidence: add decoded JWT claim block. Suggested Fix: Java snippet with `auth.roles()` check. |
| **SSRF (backchannel / CIBA)** | HTTP Evidence: Docker log line is the primary confirmation — note it explicitly. Execution Logs: add `### Docker Log Evidence` sub-section. Sandbox Configuration: include SSRF payload endpoint and SPI auth channel URI rows. |
| **Policy / grant type bypass** (ROPC, client policy) | HTTP Evidence: show the token endpoint request and response. Checklist §4 (Protocol Implementation) will have the most active items. |
| **Parameter pollution / open redirect** | HTTP Evidence: show the redirect URL and highlight the duplicate parameters. |
| **Third-party / dependency CVE (novel pattern)** | Root Cause Analysis: replace anti-pattern with "Keycloak amplifier". Suggested Fix: dependency upgrade + interim mitigations (not a Java fix sketch). Severity Assessment: note upstream CVSS and operational conditionality. Action Items: first P1 is the BOM/dependency upgrade, not a code guard. |
| **DoS / algorithmic complexity** | Sandbox Configuration: add exploit parameters rows. Execution Logs: add post-storm health check result. HTTP Evidence: show health check fail as the confirmation signal (not an HTTP 2xx/4xx on a business endpoint). |

## 7. What Must Not Vary

These elements must be **identical** across all reports regardless of issue type:

1. The H1 title format: `# Security Triage Report — Issue #{{N}}`
2. The front-matter field names and order (§2)
3. The section heading names and phase prefixes (§5)
4. The section order (§3 table)
5. The guardrails table structure with all 8 rules (§4.2)
6. The pattern match results code block listing all registered CVE patterns (§4.4.2)
7. The Session Metrics field names (§4.15) — all 14 fields must be present in every executed session report
8. The verdict JSON schema (§4.16)
9. The footer wording (§4.17)

---

## 8. Enforcement in SKILL.md / AGENTS.md

When generating a report, Bob-Sentry modes MUST:

1. Determine the session type (`plan-only`, `executed`, or `escalated`) before writing
2. Look up §3 to confirm which sections are required vs omitted
3. Use the exact heading names from §5 — never paraphrase them
4. Populate each section with real session data — no placeholder text in the final file
5. Reference this document's §6 for any issue-type-specific content variations

The report is **not complete** until:
- All `●` sections for the session type are present with real content
- The front-matter block is fully populated
- The Session Metrics table is present with all 14 fields populated (for `executed` sessions)
- The Verdict JSON is present and machine-parseable
- The footer is present
- `.bob/reports/metrics-summary.md` has been updated with this session's row (see §4.15
  dual-write rule) — the report file and metrics-summary MUST be consistent
