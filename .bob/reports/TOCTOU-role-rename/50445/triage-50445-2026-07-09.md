### Root Cause Analysis

The vulnerability is an **Authorization Bypass via Missing Role Permission Check** on the
Protocol Mapper creation endpoint:

```
Attack path:
  1. Attacker holds delegated admin token with manage-clients role
  2. POST /admin/realms/{realm}/clients/{clientId}/protocol-mappers/models
     Body: {"protocolMapper": "oidc-hardcoded-role-mapper", "config": {"role": "realm-admin"}}
  3. ProtocolMappersResource.createMapper() тЖТ managePermission.require()
     тЖТ auth.clients().requireManage(client) тЖТ PASSES (attacker has manage-clients)
  4. validateModel() тЖТ HardcodedRole.validateConfig() тЖТ base no-op тЖТ NO ROLE CHECK
  5. client.addProtocolMapper(model) тЖТ mapper persisted to database
  6. Next token issued for any user of this client contains: {"realm_access": {"roles": ["realm-admin"]}}
  7. Attacker (or any other user of the client) now has realm-admin capabilities in every token

Missing guard:
  auth.roles().requireMapRole(realmAdminRole) тЖР NEVER CALLED in mapper creation path

The correct guard exists in RolePermissions.checkAdminRoles() and
RolePermissionsV2.canMapRole() but these are wired to RoleMapperResource (role assignment
to users/groups), NOT to ProtocolMappersResource (mapper creation to clients).
```

**Why this is BLOCKER-class despite `priority/important` label:**
- The injected mapper fires for ALL users of the client, on ALL grant types, indefinitely
- A single manage-clients delegated admin can silently escalate ALL users of any client to realm-admin
- The attack is stealthy: the mapper appears as a legitimate protocol mapper configuration
- Unlike a direct role assignment (which is logged and visible in user role lists), a hardcoded
  role mapper is stored at the client level and may not be in the regular RBAC audit surface

## Phase 4 тАФ Triage Report

### Vulnerability Summary

**CVE-2026-4629** is a privilege escalation vulnerability in the Keycloak Admin REST API.
A user holding only the `manage-clients` delegated admin role can create an
`oidc-hardcoded-role-mapper` on any client, injecting `realm-admin` (or any other realm
role) into the access tokens of every user who authenticates through that client.

The `manage-clients` permission gate in
[`ProtocolMappersResource.createMapper()`](../../../services/src/main/java/org/keycloak/services/resources/admin/ProtocolMappersResource.java:137)
checks only whether the caller can manage the **client container** тАФ it never verifies
whether the caller has permission to **inject** the specific role being hardcoded.
The `HardcodedRole` mapper has no `validateConfig()` override and the role-permission
evaluation path (`auth.roles().requireMapRole()` / `checkAdminRoles()`) is wired only
to the direct role assignment endpoints in `RoleMapperResource`, not to the mapper pipeline.

### Code Coordinates тАФ Confirmed Missing Guard

```java
// ProtocolMappersResource.java:137 тАФ createMapper()
// CURRENT (VULNERABLE):
public Response createMapper(ProtocolMapperRepresentation rep) {
    managePermission.require();   // only checks manage-clients
    model = RepresentationToModel.toModel(rep);
    validateModel(model);          // calls HardcodedRole.validateConfig() тАФ no-op for role check
    model = client.addProtocolMapper(model);
    ...
}

// MISSING: for oidc-hardcoded-role-mapper, should also call:
// auth.roles().requireMapRole(resolvedRole)
// or HardcodedRole.validateConfig() should resolve the role and call
// auth.roles().requireMapRole() on it.
```

### Recommended Fix (Sketch тАФ for engineer review)

Two complementary approaches; either or both should be applied:

**Option A тАФ Enforce in `HardcodedRole.validateConfig()`:**
Override `validateConfig()` in
[`HardcodedRole.java`](../../../services/src/main/java/org/keycloak/protocol/oidc/mappers/HardcodedRole.java)
to resolve the configured role name and call `auth.roles().requireMapRole(role)` if an
`AdminPermissionEvaluator` is available in the session context. This is the targeted fix.

**Option B тАФ Enforce in `ProtocolMappersResource.createMapper()`:**
After `validateModel()`, if the mapper is `oidc-hardcoded-role-mapper`, resolve the role
from `model.getConfig().get("role")` and gate on `auth.roles().requireMapRole(resolvedRole)`.

Both options must also cover the `PUT` (update) path at line 238.
