## Executive Summary

A user holding the `manage-clients` permission in a Keycloak realm can escalate to full
`realm-admin` by injecting a **HardcodedRole protocol mapper** into any OIDC client. The
mapper type `oidc-hardcoded-role-mapper` is a built-in Keycloak mapper that embeds a
specified role claim into every token issued for the client at generation time, bypassing
all scope restrictions.

This was reproduced live in a Keycloak 26.1 sandbox:

1. A delegated admin user (`delegated-admin`) was granted only `manage-clients` — explicitly
   **not** `realm-admin`.
2. `delegated-admin` posted a HardcodedRole mapper (`role=realm-admin`) to `triage-client`.
3. Keycloak accepted the mapper with `201 Created`.
4. A subsequent ROPC token request for `triage-user` via `triage-client` returned a JWT
   containing `realm-admin` in `realm_access.roles`.

The vulnerability is fully confirmed. Any user or service account with `manage-clients`
can silently grant themselves (or any user) realm-administrator privileges.

## Root Cause Analysis

**Vulnerable code area:**

```
services/src/main/java/org/keycloak/services/resources/admin/ProtocolMappersResource.java
  → createMapper() — authorises based on manage-clients only, no check for role sensitivity
services/src/main/java/org/keycloak/protocol/oidc/mappers/HardcodedRole.java
  → setClaim() — unconditionally injects the configured role into access/id/userinfo tokens
services/src/main/java/org/keycloak/protocol/oidc/TokenManager.java
  → getAccesToken() — iterates protocol mappers including HardcodedRole without guard
```

**Root cause:**  
`ProtocolMappersResource.createMapper()` checks that the calling admin has `manage-clients`
but does **not** validate whether the mapper being created could inject roles that exceed the
caller's own privilege level. A user with `manage-clients` but not `realm-admin` can
therefore create a mapper that grants `realm-admin` — a privilege they do not hold — to
every token issued for that client. There is no "privilege ceiling" check on mapper creation.

**Architectural anti-pattern:**  
Authorization asymmetry (Class 10). The permission check on `createMapper()` validates
that the admin can manage the client, but does not validate that the mapper content
(role injection) is within the admin's own privilege scope. This is the same root cause
category as CVE-2026-11986 (FGAP unassign) — an operation is permitted to create an
effect that the actor themselves couldn't achieve directly.

## Suggested Fix Area

```java
// ProtocolMappersResource.java — createMapper()
// Add privilege ceiling check before persisting mapper:
if (mapper.getProtocolMapper().equals("oidc-hardcoded-role-mapper")) {
    String roleName = mapper.getConfig().get("role");
    RoleModel role = realm.getRole(roleName);
    if (role != null && !auth.roles().canAssign(role)) {
        throw new ForbiddenException(
            "Cannot create a mapper that injects a role beyond caller's privilege scope"
        );
    }
}
```

The check must use `auth.roles().canAssign(role)` (the same FGAP-aware check used during
direct role assignment) so the guard is consistent across all code paths — mirroring the
fix applied for CVE-2026-11986.
