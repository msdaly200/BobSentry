## Executive Summary

A delegated admin with `view-realm` permission (but explicitly without `view-clients` or
`manage-clients`) can enumerate hidden client IDs by calling the Admin UI extension endpoint
`GET /admin/realms/{realm}/ui-ext/authentication-management/flows`. When a confidential client
has a custom (non-built-in) authentication flow bound to it, the flow summary mapper populates
`usedBy.type=SPECIFIC_CLIENTS` and includes the client's `clientId` in the `values` array —
without applying the `auth.clients().canView(client)` check used by the standard client
endpoints. The vulnerability was confirmed live in a Keycloak 26.2.5 sandbox: the hidden
client ID `triage-secret-client` appeared in the flow summary response to a view-realm-only
actor, while the direct `GET /clients/{id}` path correctly returned `403 Forbidden` for the
same actor. Severity is **Medium** — requires authenticated admin token; leaks metadata
(client IDs and flow topology) rather than client secrets.

## Phase 2 — Root Cause Analysis

**Vulnerable code area:**

```
js/apps/admin-ui/src/authentication/flows/                         (Admin UI React)
services/src/main/java/org/keycloak/services/resources/admin/
    AuthenticationManagementResource.java                           (backend — /authentication/flows)
services/src/main/java/org/keycloak/services/resources/admin/ext/
    AdminExtResource.java or equivalent ui-ext handler              (Admin UI extension endpoint)
server-spi-private/src/main/java/org/keycloak/models/utils/
    RepresentationToModel.java / ModelToRepresentation.java         (flow summary mapper)
services/src/main/java/org/keycloak/services/resources/admin/fgap/
    ClientPermissions.java                                          (auth.clients().canView())
```

**Root cause:** The Admin UI extension endpoint `GET /ui-ext/authentication-management/flows`
applies only `requireViewAuthenticationFlows()` at the endpoint level. This check is satisfied
by `view-realm`. When building the `usedBy` summary for each flow, the mapper iterates all
clients with `authenticationFlowBindingOverrides` pointing to that flow and adds those client
IDs into `usedBy.values` (with `type=SPECIFIC_CLIENTS`). It does so **without calling**
`auth.clients().canView(client)` for each client in the iteration — the same permission check
used by `ClientResource.getClient()`. An actor who cannot read any client through the standard
path can enumerate all client IDs bound to custom flows via this unguarded aggregation query.
The same analysis applies to identity-provider alias inclusion — IdP aliases are added without
an analogous `auth.identityProviders().canView(idp)` check.

**Architectural anti-pattern:** Authorization asymmetry — Class 10. The permission check is
applied on the primary resource endpoint (`ClientResource`) but skipped on a secondary
aggregation endpoint that returns the same sensitive data. This is the same root cause as
CVE-2026-11986 (assign enforced, unassign skipped) but applied to read-path aggregation
instead of write-path operations.

## Phase 4 — Suggested Fix / Remediation

The fix requires adding a `canView(client)` guard in the flow summary mapper wherever it
populates `usedBy.values` with client IDs. The analogous IdP alias population also needs a
corresponding `canView(idp)` guard.

**Fix sketch — flow summary mapper:**

```java
// In the method that builds AuthenticationFlowRepresentation.usedBy
// (likely ModelToRepresentation.toRepresentation() or a dedicated FlowSummaryMapper)

// Current (vulnerable):
for (ClientModel client : realm.getClientsStream()
         .filter(c -> flowId.equals(c.getAuthenticationFlowBindingOverride("browser"))
                   || flowId.equals(c.getAuthenticationFlowBindingOverride("direct_grant")))
         .collect(Collectors.toList())) {
    usedByValues.add(client.getClientId());   // ← no permission check
}

// Fixed:
for (ClientModel client : realm.getClientsStream()
         .filter(c -> flowId.equals(c.getAuthenticationFlowBindingOverride("browser"))
                   || flowId.equals(c.getAuthenticationFlowBindingOverride("direct_grant")))
         .collect(Collectors.toList())) {
    if (auth.clients().canView(client)) {      // ← add this guard
        usedByValues.add(client.getClientId());
    }
}

// Similarly for identity providers:
for (IdentityProviderModel idp : realm.getIdentityProvidersStream()
         .filter(p -> flowId.equals(p.getFirstBrokerLoginFlowId())
                   || flowId.equals(p.getPostBrokerLoginFlowId()))
         .collect(Collectors.toList())) {
    if (auth.identityProviders().canView(idp)) {  // ← add this guard
        usedByValues.add(idp.getAlias());
    }
}
```

**Where it belongs:**
- [`services/src/main/java/org/keycloak/services/resources/admin/AuthenticationManagementResource.java`](services/src/main/java/org/keycloak/services/resources/admin/AuthenticationManagementResource.java) — the `getFlows()` method or its flow representation helper
- The `/ui-ext/authentication-management/flows` handler in the Admin UI extension layer

**Related files to audit:**
- [`services/src/main/java/org/keycloak/services/resources/admin/ext/`](services/src/main/java/org/keycloak/services/resources/admin/ext/) — Admin UI extension endpoints
- [`services/src/main/java/org/keycloak/services/resources/admin/fgap/ClientPermissions.java`](services/src/main/java/org/keycloak/services/resources/admin/fgap/ClientPermissions.java) — verify `canView()` is usable in this context without session side-effects
- Any other aggregation endpoints that return client or IdP metadata (e.g., `ClientResource.getDefaultScopes()`, scope mapping endpoints) — audit for the same missing per-resource guard pattern
