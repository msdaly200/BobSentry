## Executive Summary

A delegated user administrator holding only `view-users` permission (with no `view-clients` or `manage-clients`) in a FGAP v2-enabled realm can enumerate hidden confidential client UUIDs, client IDs, active session participation, offline-token grants, and consent metadata through three user subresource endpoints in the Keycloak Admin REST API. The FGAP per-client visibility filter is correctly applied to `GET /admin/realms/{realm}/clients/{id}` (returns 403), but is absent from `GET /users/{id}/sessions`, `GET /users/{id}/consents`, and `GET /users/{id}/offline-sessions/{clientUuid}`. Confirmed live on Keycloak latest (current development branch). This is a novel sub-variant of CVE-2026-11986 affecting a different code surface — user session/consent subresources rather than role-assignment asymmetry.

## Phase 2 — Root Cause Analysis

**Vulnerable code area:**
```
services/src/main/java/org/keycloak/services/resources/admin/UserResource.java
  — getSessions()       → GET /users/{id}/sessions
  — getOfflineSessions() → GET /users/{id}/offline-sessions/{clientUuid}
  — getConsents()       → GET /users/{id}/consents

services/src/main/java/org/keycloak/services/resources/admin/ext/AdminExtResource.java
  (or AdminRealmResource ui-ext handler)
  — session map builder  → GET /ui-ext/sessions
```

**Root cause:** `UserResource.getSessions()` builds a `UserSessionRepresentation` that includes a `clients` map (keyed by client UUID, valued by client ID string). The endpoint checks `auth.users().canView(user)` — which the `view-users` permission satisfies — but does **not** iterate over the `clients` map and apply `auth.clients().canView(client)` to filter out clients the caller is not authorized to see. The same omission applies to `getConsents()` and `getOfflineSessions()`, which construct responses that include client UUID and clientId fields without per-client visibility gating. The Admin UI extension session endpoint (`ui-ext/sessions`) has the same gap — it checks `requireViewAuthenticationFlows()` or equivalent realm-level permission but does not filter by per-client RBAC.

**Architectural anti-pattern:** Authorization asymmetry — Class 10. A high-level permission (`view-users`) grants access to a resource but the response builder includes child objects (client references) that should be filtered by a separate, narrower permission (`view-clients`). This is the same pattern as #50983 (missing `canView()` in flow summary mapper) and the original CVE-2026-11986 (role assignment without symmetric unassignment check).

## Phase 4 — Suggested Fix / Remediation

The fix must apply per-client visibility filtering inside the response builders for `UserResource`. Two strategies are acceptable:

**Option A (Preferred — filter response):** After building each `UserSessionRepresentation`, iterate over its `clients` map and remove entries where `auth.clients().canView(clientModel)` returns false. Same pattern for consent and offline-session response objects.

```java
// UserResource.java — getSessions(), getOfflineSessions(), getConsents()
// After building the representation, filter each client reference:
List<UserSessionRepresentation> filtered = sessions.stream()
    .peek(rep -> {
        Map<String, String> visibleClients = rep.getClients().entrySet().stream()
            .filter(e -> {
                ClientModel client = realm.getClientById(e.getKey());
                return client != null && auth.clients().canView(client);
            })
            .collect(Collectors.toMap(Map.Entry::getKey, Map.Entry::getValue));
        rep.setClients(visibleClients);
    })
    .collect(Collectors.toList());
```

**Option B (Stricter — require view-clients before listing):** For `getOfflineSessions(clientUuid)` specifically, require `auth.clients().requireView(client)` before returning any data for that client. This is simpler but returns 403 rather than a filtered list.

```java
// UserResource.getOfflineSessions()
ClientModel client = realm.getClientById(clientId);
auth.clients().requireView(client);  // add this guard
```
